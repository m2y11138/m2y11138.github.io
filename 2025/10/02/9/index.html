SSR: 很少的 JS 需要下载，加载页面的速度很快，数据会在 html 渲染之前 fetch 交互效果不好 因为其他的页面并不是这次获取的，而是当你进入一个 url 之后向服务器重新获取新的页面内容 因为每次获取只获取这次需要的内容 如果这次不需要那么就不获取了 SEO很好 向一个真正的网页

CSR: 客户端渲染 比较慢因为要下载完整的 JS 代码，在组件挂在完之后才开始数据 fetch 用户交互很好 是真正的网页工具 像一个 TOOL

所以做项目之前考虑 服务端渲染 还是客户端渲染是很重要的

SSR 也有两种 静态SSR和动态SSR

静态SSR并不会每次请求的时候都渲染，而是每次返回你部署的时候渲染的HTML

动态SSR服务器会在每次请求页面的时候重新渲染一个HTML返回

CSR: HTML+CSS+JS(包含项目所有的逻辑 很大) => 整合资源 => download(fetch) data / render spinner (First paint) => render app with data (Content paint)

SSR: fetch data => render app with data => html + css +js [很少的JS] [First paint] [Content paint] => hydrate (static html 添加 JS 变成有交互性的 html)

如何让静态 html 拥有 JS 效果

而不是把 JS 代码一律渲染成 string

Hydration: 水合过程 添加HTML在服务端渲染时失去的交互和事件处理功能 所以也需要发送 react js bundle 到客户端

之后 React 会构建客户端的组件树和服务端的DOM做对比 然后应用 js 效果 为了节省时间而不是创建一个新的DOM 应用现有的DOM

水合过程是服务端渲染的最后一步 成为一个真正的服务端渲染的 app 服务端无法发送交互性网页 只能发文字 浏览器也只是解析为 content-type:text/html 文字做解析

如果有位置DOM错误 就会报错 hydration error

常见的水和错误原因有: 错误的 HTML元素使用，服务端和客户端数据不同步，使用只在浏览器中的对象例如window,localstorage,side effects and so on

所以有些是客户端 JS 有些是服务端 JS

客户端 JS 要写入html
